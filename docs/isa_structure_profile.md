### 3.3 numbering 구조적 특성

**핵심 사실**:
- 문단번호(`1.`, `A1.`)는 **문서 본문에 텍스트로 저장되지 않음** — Word의 `numPr` → `numId`/`ilvl` 조합으로 자동 생성
- 각 감사기준서마다 **별도 numId**를 사용 (카운터가 개별적으로 재시작됨)
- 전체 742개 numId 중 절반 이상이 `abstractNumId=51` (A%1.) 또는 `abstractNumId=98/140` (%1.) 에 연결

**numId=0 특수 케이스**:
- 303개 문단이 `numId='0'`으로 마크됨
- 이는 이전에 번호가 붙어 있었지만 **명시적으로 제거된** 문단 (`<w:numId w:val="0"/>` = 넘버링 제거)
- 이 문단들은 `문단A` 스타일을 가지며 본문 단락임 — 번호 없는 일반 단락으로 처리해야 함

**lvlOverride**:
- 일부 numId는 `<w:lvlOverride>` + `<w:startOverride>` 를 통해 카운터를 재시작
- 기준서 경계마다 새 numId를 할당하는 방식이 주 패턴이므로 lvlOverride 빈도는 낮음

## 6. 목차(TOC) 구조

### 6.1 TOC 위치 및 범위

- **스타일**: `목차` (한국어 명칭)
- **문단 수**: 764개
- **위치**: para idx 8 ~ 10692 (전체 문서에 분산)

### 6.2 TOC 구조 특이점

- 전통적으로 TOC는 문서 앞부분에만 있어야 하지만, 이 문서에서는 **각 기준서별 소목차**가 해당 기준서 내부에도 포함됨
- 즉 TOC 스타일 단락은 문서 전반에 걸쳐 반복 등장
- 예: `목차` 스타일 문단 = `적용 및 기타 설명자료............70` 등 페이지 번호 포함 항목

### 6.3 전체 목차 vs 개별 기준서 목차

| 위치 | 내용 |
|------|------|
| para 0~4 | 표지: `회계감사기준 전문(全文)`, `목 차`, `페이지` |
| para 5 | `heading 1`: ISA-200 시작 (첫 번째 기준서 제목) |
| para 8~ | `목차` 스타일 단락 시작 — 각 기준서 내부 목차 항목들 |

**결론**: TOC/본문 경계는 `heading 1` + `감사기준서 NNN` 패턴으로 탐지하는 것이 가장 신뢰성 있음. `목차` 스타일 단락은 **파싱 시 `is_toc=True`로 마킹 후 제외** 처리 필요.

## 8. Phase 1 구현 시 유의할 Edge Case

### EC-1: numId=0 처리 (번호 제거된 문단)

- **현상**: 303개 문단이 `<w:numId w:val="0"/>` — 이는 스타일 상속으로 붙은 번호를 **명시적으로 제거**한 표시
- **위험**: `numId=0`을 "numId 없음"으로 취급하면 `abstractNumId=?`가 되어 분류 오류
- **대응**: `numId='0'` 은 반드시 번호 없음으로 처리. `numId=None`(numPr 자체가 없음)과 구분 필요

### EC-2: 목차(`목차` 스타일) 문단이 본문에 혼재

- **현상**: TOC 스타일 문단이 764개로 para idx 8~10692 전역에 산재
- **위험**: heading_trail 추적 시 TOC 항목이 섹션 헤딩으로 오인될 수 있음
- **대응**: `is_toc=True` 마킹 후 heading stack 업데이트에서 제외. 단, ISA 경계 탐지 전에 TOC 필터링 적용

### EC-3: 감사보고서 예시 구간 (ISA-700~720)

- **현상**: ISA-700부터 700번대 기준서에는 실제 감사보고서 예시 텍스트가 대량 포함
- **위험**: `독립된 감사인의 감사보고서`, `감사의견`, `감사의견근거` 등이 수십 회 반복 — 이들은 본문이 아닌 **예시(illustration)**
- **대응**: `보론 제목` 스타일이 등장한 이후 or 무스타일 블록 연속 등장 시 `kind='example_text'`로 별도 분류

### EC-4: ISA-315 내부 복잡 heading 구조 (하위 기준서 참조 헤딩)

- **현상**: `감사기준서 315와 감사기준서 610 간의 관계` 가 `heading 3` 스타일로 등장 → isa_pattern 매칭 시 `ISA-315`로 오탐
- **위험**: heading stack에 `315`가 중복 push되어 section 추적 오류
- **대응**: ISA 기준서 번호 탐지 시 반드시 `heading 1` 스타일로 한정. `heading 2~9` 에서 `감사기준서 NNN` 텍스트가 나와도 경계로 취급하지 않음

### EC-5: 적용지침 `A%1.` 번호와 요구사항 `%1.` 번호의 카운터 독립성

- **현상**: 각 기준서는 독립적인 numId를 사용. `A1.`, `A2.`... 는 `abstractNumId=51`의 별도 numId 인스턴스로 관리
- **위험**: numId가 기준서마다 달라 `numId=86`이 ISA-200에서 `1.`을 의미해도 ISA-210에서는 다른 번호를 나타낼 수 있음
- **대응**: 카운터 replay 시 **기준서 경계(heading 1 변경 시점)**에서 각 추상 번호 유형별 카운터를 리셋해야 함

### EC-6: `보론 제목` 스타일의 heading_trail 처리

- **현상**: 보론은 `적용 및 기타 설명자료`의 하위이지만 `보론 제목` 스타일은 `heading` 계열이 아님
- **위험**: heading stack을 `heading N` 기반으로만 구성하면 보론 내 문단의 `heading_trail`이 `적용 및 기타 설명자료` 직속으로 오기됨
- **대응**: `보론 제목` 스타일을 `heading 3` 수준으로 취급하는 별도 규칙 필요. 보론 내부는 `section='application_guidance'` + `is_appendix=True` 플래그

### EC-7: 다중 ilvl 하위항목의 parent_paragraph_id 연결

- **현상**: 적용지침 `A%1.`(ilvl=0) 아래 `불릿목록A`(ilvl=1), `목록A`(ilvl≥1) 등 다중 수준 하위항목 존재
- **위험**: `parent_paragraph_id` 를 단순히 "직전 ilvl=0 문단"으로 잡으면, `불릿목록A`(비번호) → 요구사항 번호의 연결이 깨짐
- **대응**: heading_trail에 ilvl별 "최근 상위 번호 문단" 스택을 별도 유지. ilvl 상승 시 pop, 하강 시 push

### EC-8: ISA-1200의 66행×2열 대형 표 청킹

- **현상**: ISA-1200 내 66행×2열 표 — 단일 청크로 만들면 4000 토큰 초과 가능성 높음
- **위험**: 표 단위 청킹 전략이 정해지지 않으면 embedding 실패 or 품질 저하
- **대응**: Phase 2 전에 표 청킹 전략 결정 필수. 제안: 행 그룹별 분할(최대 10~15행) + heading_trail 재사용

### EC-9: 감사보고서 예시의 placeholder 텍스트

- **현상**: `[감사보고서일]`, `[감사인의 주소]`, `ABC주식회사의 주주 귀중` 등이 각각 41회, 41회, 29회 등장
- **위험**: 이 텍스트들이 본문 청크에 포함되면 임베딩 품질 저하 및 오검색 유발
- **대응**: `is_example=True`로 마킹하거나 별도 chunk kind 할당. Phase 2 스키마에 `kind='example'` 추가 검토

### EC-10: `문단A` vs `문단` 스타일 혼용

- **현상**: `문단A` (1916개), `문단` (1316개) — 두 스타일이 구분 없이 혼용
- **위험**: 둘 다 본문 단락이지만 포맷이 다를 수 있어 kind 매핑 시 중복 처리 필요
- **대응**: 두 스타일 모두 `kind='paragraph'`로 단일화. 렌더링 차이는 무시. 추후 `_IFRS_parsing` 비교 후 재평가

## 10. Phase 1 구현 권장사항

### 10.1 감사기준서 경계 탐지

```python
# 신뢰성 있는 경계: heading 1 + 감사기준서 패턴
import re
ISA_BOUNDARY = re.compile(r'^감사기준서\s+(\d{3,4})')
if style == 'heading 1' and ISA_BOUNDARY.match(text.strip()):
    # 새 기준서 시작
```

### 10.2 numId 해석 순서

1. `numId` 값 읽기 → `num_map[numId]` → `abstractNumId`
2. `abstractNumId` → `abstract_nums[abstractNumId][ilvl]` → `lvlText`, `numFmt`
3. `lvlText`에 `A%1.` 포함 → 적용지침, `%1.` + decimal → 요구사항
4. `numId='0'` (문자열 '0') → 번호 없음 단락

### 10.3 섹션 매핑 테이블

| 텍스트 | heading 스타일 | section 값 |
|--------|:---:|:---|
| `서론` | heading 2 | `intro` |
| `감사인의 전반적인 목적` | heading 2 | `overall_objectives` |
| `목적` | heading 2 | `objectives` |
| `용어의 정의` | heading 2 | `definitions` |
| `요구사항` | heading 2 | `requirements` |
| `적용 및 기타 설명자료` | heading 2 | `application_guidance` |
| (보론 제목) | `보론 제목` | `appendix` |

### 10.4 abstractNumId 핵심 2종

```
요구사항 번호: abstractNumId ∈ {70, 98, 140, ...} → lvlText='%1.', numFmt='decimal'
적용지침 번호: abstractNumId ∈ {15, 51} → lvlText='A%1.', numFmt='decimal'
```

이 두 유형을 기준으로 `kind='requirement'` vs `kind='application_guidance'` 분류.

### 11.1 Style 레벨 numPr 상속 (§3 에 누락되어 있었음)

**핵심 누락**: §3.1~§3.4 는 `word/numbering.xml` 과 문단 직접 `<w:pPr><w:numPr>` 만 기술했으나, 실제 ISA 2025 DOCX 는 **`word/styles.xml` 의 style 레벨에서 numPr 이 상속**되는 구조가 주된 메커니즘이다.

#### 본 문서의 style→num 기본값 (word/styles.xml 실측)

| styleId | 한글명 | based_on | 상속되는 numId | 상속되는 ilvl | 의미 |
|---|---|---|---|---|---|
| `a1` | 문단 | — | `119` | 0 | 요구사항 `1.`, `2.` ... |
| `A` | 문단A | — | `105` | 0 | 적용지침 `A1.`, `A2.` ... |
| `A0` | 불릿목록A | — | `105` | 1 | 적용지침 하위 `(1)`, `(2)` |
| `A2` | 목록A | — | `119` | 1 | 요구사항 하위 `(1)`, `(2)` |
| `B` | — | `A0` | `105` | 2 | 적용지침 하위 `(가)`, `(나)` |
| `B0` | — | `A2` | `119` | 2 | 요구사항 하위 `(가)`, `(나)` |

#### 실측 결과의 결정적 함의

- ISA-200 의 요구사항 `1.`~`29.` · 적용지침 `A1.`~`A82.` 는 **전부 style 상속** 으로 번호 부여.
- ISA-1200 의 요구사항 `1.`~`152.` 도 **전부 style 상속**.
- 문단 직접 numPr 은 이들에 대해 부재 (`<w:pPr>` 에 `<w:pStyle w:val='a1'/>` 만 있음).
- 따라서 파서는 반드시 다음 체인으로 `(numId, ilvl)` 를 해결해야 한다:

```
문단 <w:pPr><w:numPr>  >  style numPr  >  style basedOn 체인 (재귀, max depth 10)
```

세부 구현 사양은 `docs/numbering_strategy.md` §9.2~9.4 addendum 참조.

### 11.2 ISA-1200 특이구조 정정 (§2.2 정정)

§2.2 는 "ISA-1200 은 `목적` 섹션이 3회 등장" 으로 기술했으나, **실제 heading 2 레벨 구조는 다음과 같다** (raw DOCX 및 parsed MD 대조):

| heading 2 | 역할 | 제안 section enum |
|---|---|---|
| 서론 | 공통 | `intro` |
| **일반원칙과 책임** | ISA-1200 고유 | `general_principles` |
| **감사업무의 수임 또는 유지** | ISA-1200 고유 | `engagement_acceptance` |
| **감사 계획** | ISA-1200 고유 | `planning` |
| **위험평가** | ISA-1200 고유 | `risk_assessment` |
| **평가된 위험에 대한 감사인의 대응** | ISA-1200 고유 | `risk_response` |
| **결론 및 보고** | ISA-1200 고유 | `conclusion_reporting` |
| 보론 1 용어의 정의 | 공통 (부록) | `appendix` |
| 보론 2 감사보고서 | 공통 (부록) | `appendix` |

총 **9 개 heading 2** 중 **6 개가 ISA-1200 고유** section 이다. §2.1 표의 `| 1200 | O | - | O | O | - | - | - |` 에서 `목적` O 표시는 **오류**로 보이며, 정정해야 한다(본 ISA 는 `목적` heading 2 를 갖지 않고 `일반원칙과 책임` 이하 custom heading 으로 구성됨).

> ISA-1200 은 `요구사항` / `적용 및 기타 설명자료` 의 대립 구조를 갖지 않고, 감사 수명주기(수임→계획→위험평가→대응→보고) 축을 따라 재편된 특수 기준서다. `structure.py` 는 SECTION_ENUM 을 위 6 개로 확장 필요.

### 11.3 §10.3 heading→section 매핑 확장

§10.3 은 "표준 5 섹션 + appendix" 기반 매핑만 제시했다. ISA-1200 지원을 위해 다음을 추가해야 한다:

| heading 2 텍스트 | section enum |
|---|---|
| 일반원칙과 책임 | `general_principles` |
| 감사업무의 수임 또는 유지 | `engagement_acceptance` |
| 감사 계획 | `planning` |
| 위험평가 | `risk_assessment` |
| 평가된 위험에 대한 감사인의 대응 | `risk_response` |

`structure.py` 의 section enum 정의에 위 6 개를 추가하고 heading2 → section 매핑 딕셔너리를 확장한다. ISA-200 의 `감사인의 전반적인 목적` 은 기존 `overall_objective` 를 그대로 사용.
